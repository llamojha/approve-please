## Why this matters

The product brief promises more than the code delivers, and this is the largest gap between stated vision and implementation. `.kiro/steering/product.md` says: *"Learning mode ‚Äî Teaches players to spot real bug patterns ‚Ä¶ PRs are curated to build skill progressively."* In code, `learning` does exactly two things: disables queue-aging meter drain (`getQueueAgingDelta` in `context/GameContext.tsx:142‚Äì163` returns `null` for learning) and splits the leaderboard by mode. PR selection is identical in both modes: `hooks/usePRSpawner.ts` draws from one shuffle bag of all preference-matching templates with no ordering by difficulty, day, or bug type. Nothing is "curated" and nothing "builds progressively." The content is ready for a curriculum ‚Äî 400 templates already tagged with `bugPatterns[].kind` (`logic|security|performance|style`) and `severity` (`minor|major|critical`), with clear easy‚Üíhard bands visible in the folder naming (e.g. `css-001-basic-color-styling` ‚Üí `css-080-buggy-transform`, syntax-error JS templates vs subtle-logic ones) ‚Äî but no code reads any of it for sequencing.

## Current state

- `hooks/usePRSpawner.ts` ‚Äî one shuffle bag: `basePool = prTemplates.filter(templateMatchesPreference)` (lines 160‚Äì169), reshuffled when exhausted (`drawBatch`, lines 97‚Äì128). `difficulty` is not even read by this hook.
- `context/GameContext.tsx` ‚Äî `difficulty: Difficulty` (`'normal' | 'learning'`) lives in game state; learning's only gameplay effect is the queue-aging skip.
- Template schema (`types/index.ts`, `PullRequestTemplate`): `templateId, importance, title, description, tags, author, files[], bugPatterns[] { kind, lineNumbers, severity, description, localizedDescription? }, localized?`. **No difficulty field exists today.**
- Content shape: `data/prTemplates/{css,generic,java,javascript,python,rust,typescript}/<slug>/template.json`; the manifest is auto-generated by `npm run generate:templates` (never hand-edit; `agent.md` Non-Negotiable #3).
- Tutorial invariant (`agent.md`): day 1 always opens with the two scripted generic PRs; waves fire at minutes 0/60/180/360. A learning curriculum must layer *selection*, not break *pacing*.
- Docs to honor: `docs/pr-template-guide.md` (template authoring guide ‚Äî extend it if the schema grows), `.kiro/steering/product.md` (the vision being served).

## Steps

### Step 1: Inventory the content by inferred difficulty

Write a small throwaway script (run with `node`, don't commit it) that walks `data/prTemplates/**/template.json` and tabulates per language: count, bugPatterns kinds/severities, line counts, and clean (no-bug) ratio. Paste the resulting table into the design doc. This grounds the curriculum bands in the actual library instead of guesses.

**Verify**: the table in the doc covers all 7 folders with non-zero counts.

### Step 2: Write `docs/learning-mode-curriculum.md`

Required sections:
1. **Goal** ‚Äî quote the product.md sentence being implemented.
2. **Proposed difficulty model** ‚Äî recommend: an explicit optional `difficultyTier: 1|2|3` field on templates, with an *inferred fallback* (derivable signals: has bugs at all, severity mix, diff line count, syntax-error-class vs subtle-logic-class) so the feature ships before all 400 files are annotated. Specify the inference rules precisely.
3. **Progression rules** ‚Äî day N of learning mode draws from which tiers with which weights; when clean PRs mix in; how repetition of a missed bug kind works (e.g. a missed `security` bug increases the weight of security templates next day ‚Äî state the exact mechanism).
4. **Pacing interaction** ‚Äî how selection layers under the untouched wave system (selection changes *what* `drawBatch` picks, never *when* waves fire).
5. **Open questions for the maintainer** ‚Äî e.g. should learning mode be endless or a fixed ladder; should it surface explanations after every miss (templates already carry `bugPatterns[].description`); does tier metadata belong in template.json or a sidecar file.
6. **Migration & effort estimate** ‚Äî what the full build needs (schema addition, generator validation, annotation pass, spawner integration, tests), with coarse S/M/L per piece.

**Verify**: doc exists and contains all six section headings (`grep -c '^## ' docs/learning-mode-curriculum.md` ‚â• 6).

### Step 3: Prototype the ordering function

`utils/curriculum.ts`: export `orderForLearning(templates: PullRequestTemplate[], day: number): PullRequestTemplate[]` implementing the doc's inferred-tier rules (pure function ‚Äî no randomness inside; take a shuffled input). In `usePRSpawner.ts`, in the base-pool effect, add a `// SPIKE:`-commented branch: when `difficulty === 'learning'`, pass the pool through `orderForLearning(pool, currentDay)` and **don't reshuffle on exhaustion** (working pool refills in the same order ‚Äî note this in the comment as a known spike simplification). You'll need `difficulty` from `useGameState()` (it's in state; just destructure it) and to add it to the effect deps.

**Verify**: `npm run typecheck && npm test` green. Manual: two learning-mode playtests reach day 2+ with visibly easier PRs (tier-1: doc/config templates and blatant syntax errors) on day 1 than the same days in normal mode.

### Step 4: Unit-test the ordering

`tests/curriculum.test.ts`: given handcrafted minimal templates spanning the inference signals, assert tier assignment matches the doc's rules and that day-1 output contains only tier-1 (or the doc's stated mix). Follow plan-001 test structure.

**Verify**: `npm test` ‚Üí all pass.
