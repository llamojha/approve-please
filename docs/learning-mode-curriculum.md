# Learning-Mode Curriculum — Design Spike (Plan 010)

Status: **design spike**. This document is the deliverable; the accompanying
prototype (`utils/curriculum.ts` + the `// SPIKE:` branch in
`hooks/usePRSpawner.ts`) is throwaway-quality, gated entirely behind
`difficulty === 'learning'`, and exists only to validate that the rules below
produce a sensible day-1-onwards experience. Nothing here changes normal mode.

## Goal

`.kiro/steering/product.md` promises:

> **Learning mode** — Teaches players to spot real bug patterns (SQL
> injection, off-by-one errors, missing tests, performance anti-patterns)
> across multiple programming languages. PRs are curated to build skill
> progressively.

Today, `learning` only (a) disables queue-aging meter drain
(`getQueueAgingDelta` in `context/GameContext.tsx`) and (b) splits the
leaderboard by mode. PR selection is identical in both modes: one shuffle bag
of preference-matching templates. Nothing is curated and nothing builds
progressively. This spike designs the curriculum that closes that gap.

Overall: **T1 = 156, T2 = 139, T3 = 95** — three usable bands. Notes:

- Diff size is a *weak* signal here: every template adds between 1 and 21
  lines, so line count cannot separate tiers on its own. Severity and bug
  *kind* are the load-bearing signals.
- The bands are healthy at the whole-library level, but skewed per language:
  `javascript` has only 5 tier-1 templates, `generic` has no tier-2. A
  single-language preference can therefore produce thin bands; the
  progression rules below define the fallback.

## Proposed difficulty model

**Recommendation: an explicit, optional `difficultyTier?: 1 | 2 | 3` field on
`PullRequestTemplate`, with a precisely-specified inferred fallback.** The
explicit field always wins when present; the fallback lets the feature ship
before any of the 390 files are annotated, and remains the permanent default
for future community templates that omit the field.

Tier semantics:

- **Tier 1 — foundations**: practice the approve/reject decision itself.
  Clean PRs and blatantly cosmetic problems (docs, config, style nits).
- **Tier 2 — visible bugs**: real bugs a careful reader can spot from the
  diff alone (broken logic, wrong comparisons, style violations with
  functional impact).
- **Tier 3 — subtle/dangerous bugs**: bugs that require domain knowledge to
  recognize (injection, leaked secrets, N+1 queries, quadratic loops) or that
  carry critical severity.

### Inference rules (exact, ordered — first match wins)

For a template `t` with `bugs = t.bugPatterns`:

1. `bugs.length === 0` → **tier 1** (clean PR; teaches confident approval).
2. Every `bugs[i].severity === 'minor'` → **tier 1** (cosmetic-only rejects).
3. Some `bugs[i].severity === 'critical'` → **tier 3**.
4. Some `bugs[i].kind ∈ {security, performance}` → **tier 3** (the "subtle
   class": these kinds require recognizing a pattern, not just reading the
   diff; the inventory confirms they cluster in the harder hand-authored
   templates).
5. Otherwise (max severity `major`, kinds ⊆ {logic, style, accessibility}) →
   **tier 2**.

Signals considered and rejected:

- **Diff line count** (`files[].lines` with `isNew`): rejected as a tier
  discriminant — the observed range (1–21 added lines) is too narrow to
  separate anything (a first-pass rule using `addedLines` thresholds put 0
  templates in tier 3). Kept only as a documented future tie-breaker if the
  library ever grows large-diff templates.
- **Tags** (`syntax`, `typo`, `docs`, …): tags are flavor text, not schema;
  28 templates carry `syntax` but tagging is inconsistent across folders.
  Not used by inference; annotation (the explicit field) is the right place
  to encode "blatant syntax error" vs "subtle logic".

The prototype implements exactly rules 1–5 in `inferTier()`
(`utils/curriculum.ts`), unit-tested in `tests/curriculum.test.ts`.

## Progression rules

Learning-mode day `N` draws from the tiers with these weights (fractions of
each draw):

| Day | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| 1 | 100% | 0% | 0% |
| 2 | 70% | 30% | 0% |
| 3 | 40% | 50% | 10% |
| 4 | 20% | 50% | 30% |
| 5+ | 10% | 40% | 50% |

- **Day 1** is tier-1 *only* (on top of the untouched scripted tutorial
  wave). If the player's language preference yields zero tier-1 templates,
  fall back to the full pool ordered easiest-first (tier ascending) rather
  than starving the waves — waves must always fire (`agent.md`
  Non-Negotiable #1).
- **Days 5 and beyond clamp** to the day-5 row: learning mode stays hard but
  never exceeds the library's ceiling.
- **Clean PRs** are not a separate track — they live inside tier 1, so the
  weights above naturally keep approval practice in the mix through day 4.
  For day 5+, the full feature should additionally enforce a **clean floor**:
  if the last 10 draws contained no clean template, force the next draw from
  the clean sub-pool. (Rationale: at 10% tier-1 weight, clean PRs get scarce
  and players drift into reject-everything heuristics.) *Not in the
  prototype.*
- **Missed-bug repetition** (the "remediation loop"): during a day, count
  per-kind misses — a *miss* is approving a PR whose template has a
  `bugPatterns[].kind = K`, or rejecting it with the wrong lines highlighted.
  At the next morning briefing, compute a per-kind multiplier
  `m(K) = min(1 + 0.5 × misses(K), 2.5)` and multiply the selection weight of
  every template containing kind `K` by `m(K)` for that day. Multipliers
  reset each day (only yesterday's misses count), so one bad security day
  yields one security-heavy day, not a permanent skew. *Not in the
  prototype* — requires miss tracking in `GameContext` state.

### What the prototype implements

The eventual feature should use **weighted sampling without replacement**
per draw. The spike approximates this deterministically: `orderForLearning`
buckets a pool by inferred tier and interleaves buckets with a smooth
weighted round-robin using the day's weights (zero-weight tiers drain last),
so any prefix of the ordered pool approximates the day's mix. `drawBatch`
then consumes the pool front-to-back as it always has. Known spike
simplifications: no randomness inside the ordering (the pool arrives in
manifest order, so repeat runs of a day see the same sequence), no reshuffle
on pool exhaustion (refills repeat the same order), no clean floor, no
remediation loop.

## Pacing interaction

The curriculum layers **selection under the untouched wave system**: it
changes *what* `drawBatch` picks, never *when* waves fire.

- Wave timing is untouched: default waves at minutes 0/60/180/360, minute 0
  always 2 PRs, weighted counts for the rest, hourly empty-queue safety
  check — all exactly as in normal mode (`agent.md` core loop).
- The day-1 scripted tutorial wave is untouched: `spawnSpecific` resolves
  templates by ID from the eager pack and never touches the pool, so the two
  scripted PRs still open day 1 before anything else.
- The only hook point is the pool itself: `applyPools` in the spawner's
  base-pool effect builds `basePoolRef`/`workingPoolRef` (once synchronously
  from eager templates, again when lazy language packs resolve). In learning
  mode the working pool is ordered by `orderForLearning(pool, currentDay)`
  instead of shuffled; in normal mode the code path is byte-identical to
  pre-spike.
- Queue-aging stays disabled in learning mode (existing behavior,
  unchanged) — the curriculum controls difficulty through content, not
  meter pressure.

## Open questions for the maintainer

1. **Endless ladder or fixed course?** Day 5+ currently clamps to the
   hardest mix forever. Alternative: a fixed N-day "course" with a
   completion screen (and the leaderboard measuring course score). Which
   matches the product vision?
2. **Post-miss explanations?** Templates already carry
   `bugPatterns[].description` (+ `localizedDescription`). Should learning
   mode show the explanation immediately after every miss (interrupting
   flow, maximal teaching) or only in the end-of-day summary (current
   incident report already lists some of this)?
3. **Where does tier metadata live?** In `template.json` (self-contained,
   travels with the template, validated by the generator) vs a sidecar file
   (e.g. `data/curriculum/tiers.json`, keeps 390 template files untouched
   and lets tiers be retuned in one place). This spike leans `template.json`
   (`difficultyTier` next to `importance`), but the annotation diff is 390
   files either way someone must review.
4. **Thin per-language bands**: with a `javascript`-only preference there
   are 5 tier-1 templates; day 1 will repeat them. Acceptable, backfill from
   `generic` regardless of preference, or author more easy templates?
5. **Should the remediation multiplier be visible?** ("Yesterday you missed
   2 security bugs — today's queue leans security.") Transparent coaching vs
   silent curation.
6. **Does learning mode ever re-enable meter pressure** at high days (e.g.
   queue aging returns on day 7+) as a bridge toward challenge mode?
